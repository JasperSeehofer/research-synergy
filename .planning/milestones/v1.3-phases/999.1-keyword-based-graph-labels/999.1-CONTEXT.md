# Phase 999.1: Keyword-Based Graph Labels - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace author/year node labels with TF-IDF keyword pills so users can visually identify paper topics at a glance. Two tiers: (1) per-node keyword pills showing top 2 keywords with score-based opacity, (2) cluster-level floating labels with dashed convex hull borders showing dominant topic per region. Includes a label mode dropdown (Keywords / Author-Year / Off) in graph controls.

</domain>

<decisions>
## Implementation Decisions

### Label Content & Display
- **D-01:** Label mode controlled via dropdown in graph controls sidebar (Keywords / Author-Year / Off). Dropdown is extensible — future categories can be added, and potentially linked to force layout clustering.
- **D-02:** Keyword mode shows top 2 TF-IDF keywords per node as separate pills with individual score-based opacity.
- **D-03:** Author-Year mode preserves current `"{first_author} {year}"` label behavior.
- **D-04:** "Off" mode hides all node labels entirely.
- **D-05:** Hover tooltip extended to show all top-5 keywords with scores, plus author/year and title. Extends existing tooltip rendering.

### Unanalyzed Paper Fallback
- **D-06:** In keyword mode, unanalyzed papers show a dimmed "[not analyzed]" badge instead of keyword pills. Visually distinguishable from analyzed nodes.

### Cluster-Level Topic Labels
- **D-07:** Clusters determined by k-means on node positions. k auto-selected via heuristic from node count (e.g. sqrt(n/2), clamped 3-8).
- **D-08:** Cluster labels are zoom-dependent (map-like LOD): visible when zoomed out, hidden when zoomed in where per-node labels take over. Works with existing LOD system.
- **D-09:** Live recompute — clustering reruns periodically during force simulation, not just after settling.
- **D-10:** Cluster visual: bordered region with dashed convex hull around cluster nodes, label pill at centroid showing dominant keyword.

### Keyword Pill Visual Style
- **D-11:** Separate pills per keyword (not comma-joined). Each pill has independent opacity based on its TF-IDF score.
- **D-12:** Score-based opacity — higher TF-IDF score = more prominent pill. Base color uniform (monochrome), differentiation via opacity only. Color-based topic coding deferred to Phase 999.2.

### Claude's Discretion
- Zoom threshold for LOD transition between cluster labels and per-node labels
- K-means recompute frequency during live simulation (balance between responsiveness and CPU cost)
- Specific opacity range mapping (e.g. score 0.3-0.9 mapped to opacity 0.4-1.0)
- Convex hull padding amount around cluster nodes
- Cluster label pill font size and background style

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `draw_label_pill()` in `label_collision.rs` — rounded pill rendering with collision avoidance. Can be reused for keyword pills with opacity parameter.
- `LabelCache` — text width measurement and collision culling system. Needs extension for multi-pill layout.
- `build_text_widths()` — offscreen canvas text measurement. Needs update for keyword text.
- Existing LOD system in `graph.rs` — progressive-reveal for 1000+ node graphs, can gate cluster label visibility.
- Convergence badge / alpha full-stop detection — can trigger cluster recompute.

### Established Patterns
- `NodeState` holds per-node rendering data, constructed from `GraphNode` in layout_state.rs
- `GraphNode` serialized from server function, deserialized in WASM — new fields cross the SSR/WASM boundary
- Label collision uses priority ordering (seed first, then citation count) — keyword pills should follow same priority system

### Integration Points
- `GraphNode` struct needs `top_keywords: Vec<(String, f32)>` or similar field
- `get_graph_data()` server function needs to join paper data with analysis data from DB
- `NodeState::label()` needs to become mode-aware (keyword vs author-year vs off)
- Graph controls sidebar needs dropdown component for label mode selection
- K-means clustering module needed — new code, no existing clustering in codebase

</code_context>

<specifics>
## Specific Ideas

- User envisions the label mode dropdown as extensible — future modes could include title, arXiv ID, etc.
- User wants label mode potentially linked to force layout clustering in the future (e.g. cluster by keyword similarity, not just spatial proximity)
- Cluster borders should use convex hull (tight-fitting polygon), not bounding boxes
- "Not analyzed" badge should be visually distinct (dimmed) so users can tell which papers need analysis at a glance

</specifics>

<deferred>
## Deferred Ideas

- **Force-layout clustering by keyword similarity** — linking label mode dropdown to actual force computation (e.g. attract nodes with shared keywords). Would be its own phase.
- **Topic-based color coding** — each keyword gets a consistent color across all nodes. Already planned as Phase 999.2 (Topic Ring Node Borders).
- **Additional label modes** (title, arXiv ID, DOI) — extensible via the dropdown, but not in scope for this phase.

</deferred>
