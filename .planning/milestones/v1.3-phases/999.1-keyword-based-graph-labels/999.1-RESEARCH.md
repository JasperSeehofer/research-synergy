## Summary

This phase layers keyword visualization on top of the existing Canvas 2D label rendering system. The keyword data already exists in the database via `PaperAnalysis.top_terms`/`top_scores`, computed by the TF-IDF pipeline. The work is primarily plumbing (extend `GraphNode` to carry keyword data from server to WASM) and rendering (draw multiple pills with varying opacity, convex hull cluster borders, cluster centroid labels).

The most architecturally significant new piece is the k-means clustering module. All other work extends existing patterns: `draw_label_pill()` already handles pill rendering and only needs an opacity parameter; `LabelCache` and `build_label_cache()` need extension for multi-pill layout; `GraphControls` needs a dropdown; `get_graph_data()` needs a DB join.

The "live recompute during simulation" requirement (D-09) is the main performance risk. K-means at 60fps on hundreds of nodes is feasible if recompute is throttled (e.g., every N frames or only when node positions change beyond a threshold) and k is kept small (clamped 3-8). The convex hull algorithm is O(n log n) for n points in a cluster — fast enough at this scale.

**Primary recommendation:** Extend the data pipeline first (GraphNode keyword field + server function join), then add the label mode enum and dropdown, then multi-pill rendering, then k-means clustering and convex hull. Ship incrementally across 4 plans in that order.

### Pattern 1: Extending GraphNode Across the SSR/WASM Boundary

**What:** `GraphNode` is serialized server-side and deserialized in WASM. New fields must implement `Serialize + Deserialize` and cross the SSR boundary cleanly.

**When to use:** Any time new per-node data must reach the frontend.

**Example (verified from existing code):**
```rust
// resyn-app/src/server_fns/graph.rs
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct GraphNode {
    pub id: String,
    // ... existing fields ...
    pub top_keywords: Vec<(String, f32)>,  // new: [(term, score), ...]
}

// In get_graph_data() SSR block — join with AnalysisRepository:
use resyn_core::database::queries::AnalysisRepository;
let analysis_repo = AnalysisRepository::new(&db);
let analyses = analysis_repo.get_all_analyses().await
    .map_err(|e| ServerFnError::new(e.to_string()))?;
let analysis_map: HashMap<String, Vec<(String, f32)>> = analyses
    .into_iter()
    .map(|a| {
        let keywords: Vec<(String, f32)> = a.top_terms.into_iter()
            .zip(a.top_scores.into_iter())
            .take(5)
            .collect();
        (a.arxiv_id, keywords)
    })
    .collect();

// Then in node construction:
GraphNode {
    // ... existing fields ...
    top_keywords: analysis_map.get(&p.id)
        .cloned()
        .unwrap_or_default(),
}
```

### Pattern 4: LabelCache Extension for Multi-Pill

**What:** `LabelCache.text_widths` currently stores one width per node (the author-year label). For keywords, each node may have 2 pills. The cache must store the total rendered width for collision testing.

**Approach:** `text_widths` stays as total width of the label footprint. For keyword mode: total = sum of pill widths + gaps. `build_text_widths()` needs the current `LabelMode` to measure the right text.

**Pitfall:** `build_text_widths()` is called once at graph load time (line 181 in graph.rs). When label mode changes at runtime, text widths must be re-measured. Add a `label_cache_dirty = true` when label mode changes (same pattern as viewport changes).

### Pattern 5: K-Means on Node Positions

**What:** Lloyd's algorithm on `(node.x, node.y)` coordinates. Pure Rust, no browser APIs needed, testable with `cargo test`.

**Recompute throttle (Claude's discretion):** Recompute every 10 simulation frames when `simulation_running` is true, and on convergence. At 60fps that's ~6 recomputes/second — visually smooth and CPU-affordable for k=3-8 on ≤500 nodes.

### Pattern 6: Convex Hull

**What:** Gift-wrapping (Jarvis march) on the node positions in a cluster. O(n*h) where h is hull size — fast for small clusters.

**Canvas drawing:** `ctx.begin_path()`, `move_to(hull[0])`, loop `line_to(hull[i])`, `close_path()`, `stroke()` with dashed line style. Must draw in world coordinates (inside viewport transform) on the label overlay canvas — or translate hull points to screen space and draw with transform=identity.

**Approach:** Draw hull on label overlay canvas (same as label pills) using screen-space coordinates (world_to_screen on each hull point), with `ctx.set_line_dash([4, 4])` for dashed style. This matches how labels are drawn — overlay canvas at CSS scale, with manual coordinate transform.

### Anti-Patterns to Avoid

- **Re-measuring text every frame:** `build_text_widths()` is expensive (browser measureText call per node). Only call it at graph load and when label mode changes. Use the dirty-flag pattern already in `RenderState`.
- **Running k-means every RAF frame:** K-means at 60fps on 500 nodes is ~30ms/frame CPU cost at worst. Throttle to every 10 frames during simulation, recompute on convergence.
- **Convex hull in world coordinates then drawing with viewport transform active:** The label overlay canvas currently clears and redraws at CSS coordinates (set_transform(dpr, 0, 0, dpr, 0, 0)). Keep hull drawing consistent with this — transform hull points with `world_to_screen()` before drawing.
- **Storing `Vec<(String, f32)>` in `NodeState` as owned data duplicated from `GraphNode`:** Move `top_keywords` into `NodeState` and populate it in `from_graph_data()`. It crosses the SSR boundary once, then lives in the WASM render state. This is the correct pattern (matches how `first_author`, `year`, etc. are handled).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| K-means for small k | Custom cluster algo | Lloyd's algorithm (~40 lines inline) | This IS hand-rolling — it's the right choice here; no crate needed for k=3-8 |
| Convex hull for small point sets | Custom geometry | Jarvis march (~60 lines inline) | Same reasoning — crates like `geo` add 200KB+ to WASM binary |
| Text width measurement | Own measureText wrapper | `build_text_widths()` already exists | Extend the existing function, don't duplicate |
| Pill rendering | Custom rounded-rect | `draw_label_pill()` already exists | Just add opacity parameter |
| Opacity animation | Custom interpolation | Direct `set_global_alpha()` per pill | Canvas API native, zero overhead |

**Key insight:** This phase is almost entirely about connecting existing pieces. The TF-IDF data exists. The pill renderer exists. The LOD system exists. The main genuinely new code is k-means + convex hull (both small, pure-Rust, testable) and the label mode enum.

## Common Pitfalls

### Pitfall 1: Text Width Cache Invalidation on Mode Change

**What goes wrong:** Label mode changes from Author-Year to Keywords, but `text_widths` still contains widths for the old label text. Collision avoidance uses wrong widths — pills overlap or are incorrectly culled.

**Why it happens:** `build_text_widths()` is called once at graph load (line 167-182 in graph.rs). It calls `n.label()` on each node, which returns the author-year string. When label mode changes, the widths are stale.

**How to avoid:** When label mode changes (detected in RAF loop by comparing previous mode to current signal value), set `label_cache_dirty = true` AND re-measure `text_widths`. Store `text_widths` as mutable in `RenderState` so the RAF loop can update it.

**Warning signs:** Pills overlapping more than expected, or good labels being culled that shouldn't be.

### Pitfall 2: Cluster Label Visible When Per-Node Labels Are Active (LOD Inversion)

**What goes wrong:** At high zoom (scale > 1.0), per-node labels are fully visible. Cluster labels (which summarize a region) become redundant and visually noisy at this zoom level.

**Why it happens:** D-08 says cluster labels are map-like LOD: visible when zoomed out, hidden when zoomed in. If both render simultaneously, the UI becomes cluttered.

**How to avoid:** Use the existing LOD constants in `lod.rs`. Recommended transition: cluster labels visible when `scale < LOD_LEVEL_1` (0.6), hidden when `scale >= LOD_LEVEL_1`. Per-node keyword pills visible at all scales (collision avoidance limits clutter anyway).

**Warning signs:** Both cluster pills and per-node pills showing simultaneously at medium zoom.

### Pitfall 3: K-Means With All Nodes at Same Position (Graph Not Yet Spread)

**What goes wrong:** At graph load before force simulation has run, all nodes start near the origin (ring placement gives some spread, but early frames are clustered). K-means assigns all nodes to one centroid or produces degenerate results.

**Why it happens:** K-means initializes centroids randomly from the node set. If nodes are tightly clustered, centroid spread is minimal and the algorithm converges to trivial k=1 clusters.

**How to avoid:** Defer first cluster computation until `simulation_settled` is true (or `alpha < 0.05` — simulation mostly settled). Skip clustering and cluster label rendering until then.

**Warning signs:** All nodes assigned to cluster 0 at startup.

### Pitfall 4: Convex Hull Drawn Under Viewport Transform

**What goes wrong:** Hull is drawn with `viewport.apply(&ctx)` active (world coordinates). But the label overlay canvas uses a separate transform (`set_transform(dpr, 0, 0, dpr, 0, 0)` at CSS resolution). Drawing hull under the wrong transform puts it in the wrong position.

**Why it happens:** The main canvas has viewport transform applied by the renderer. The label overlay canvas has a DPR scale transform only. The hull must be drawn on the label overlay in screen space.

**How to avoid:** Transform each hull vertex with `world_to_screen(x, y)` before drawing, then draw with the label overlay's DPR transform active. Consistent with how existing labels are drawn (see graph.rs line 607: `ctx.set_transform(dpr, 0.0, 0.0, dpr, 0.0, 0.0)`).

### Pitfall 5: `top_keywords` Field Missing from `NodeState` Test Helpers

**What goes wrong:** Tests for `layout_state.rs` (and `label_collision.rs`) use `make_test_node()` helper functions that construct `NodeState` directly. After adding `top_keywords` to `NodeState`, all these helpers fail to compile.

**Why it happens:** Rust struct construction is exhaustive — adding a field without a `Default` impl or updating all constructors breaks existing tests.

**How to avoid:** Add `top_keywords: Vec<(String, f32)>` to `NodeState` with `#[derive(Default)]` on `NodeState`, or add `top_keywords: vec![]` to every test helper. The latter is safer (explicit, no silent default). There are ~20 such helpers across `label_collision.rs`, `layout_state.rs`, and `lod.rs` tests.

### Pitfall 6: `get_graph_data()` Performance — N+1 Query

**What goes wrong:** Joining papers with analyses in `get_graph_data()` using `get_analysis()` per paper in a loop creates N+1 DB queries.

**Why it happens:** `AnalysisRepository` exposes `get_analysis(id)` and `get_all_analyses()`. Calling the per-paper variant in a loop is the easy path.

**How to avoid:** Use `get_all_analyses()` once and build a `HashMap<String, PaperAnalysis>`. Then look up per paper from the map during node construction. This is exactly how the existing code handles gap findings (line 151-172 of graph.rs).

## Open Questions

1. **Keyword text width measurement for multi-pill collision**
   - What we know: `build_text_widths()` measures one string per node using `n.label()`. For keyword mode, we need the total width footprint of 2 pills + gap.
   - What's unclear: Whether to refactor `build_text_widths()` to accept a closure/mode, or store a separate `keyword_text_widths` alongside `text_widths` in `RenderState`.
   - Recommendation: Add `keyword_text_widths: Vec<f64>` as a second field in `RenderState`. Compute both at graph load. Re-measure both when label mode changes. Simple, no abstraction needed.

2. **K-means centroid initialization strategy**
   - What we know: Random initialization from node positions can produce poor clusters if nodes are unevenly distributed.
   - What's unclear: Whether k-means++ seeding is worth the extra complexity.
   - Recommendation: Use k-means++ (pick first centroid randomly, then each subsequent centroid with probability proportional to distance squared from nearest existing centroid). Adds ~15 lines; meaningfully reduces degenerate clustering. Deterministic seed from node count for reproducibility.

3. **Convex hull point ordering for canvas path**
   - What we know: Canvas `lineTo` requires points in a consistent winding order.
   - What's unclear: Whether to use clockwise or counter-clockwise.
   - Recommendation: Jarvis march naturally produces counter-clockwise hull for standard coordinate system. Canvas fills and strokes work with either; use CCW by convention.
