## Summary

Phase 999.2 adds colored arc segments to each node's border ring, encoding each paper's top-3 TF-IDF keywords as a visual "topic fingerprint." The implementation spans four distinct concerns: (1) Canvas2D arc rendering on the existing overlay canvas, (2) a corpus-level palette computed server-side and cached in SurrealDB, (3) Leptos UI additions (toggle + legend) in the existing `GraphControls` component, and (4) a keyword-based dim filter that integrates with the existing `is_dimmed` logic.

The key architectural insight confirmed by code inspection: the Canvas2D overlay canvas (`label_canvas_ref`) is already used per-frame in `graph.rs` for labels and cluster hulls, cleared each frame. Topic arcs must be drawn there — after the WebGL2 node fill pass but before label pills — using `ctx.arc()` with appropriate `save()`/`restore()` wrapping for each node. The border stroke in `canvas_renderer.rs` (line 252-254) currently draws a 1px neutral stroke using `stroke()` on the already-open arc path from the fill; this pattern must be replaced for above-threshold nodes with the multi-arc approach.

The `GraphData` struct needs a new `palette: Vec<PaletteEntry>` field carrying corpus keyword→color mapping across the SSR boundary. The `get_graph_data()` server function will compute or load this palette. `GraphState` similarly needs a `palette` field for per-frame arc color lookup. No new crates are required — `web_sys::CanvasRenderingContext2d` already exposes `arc()`, `set_line_dash()`, and `set_global_alpha()`.

**Primary recommendation:** Draw topic arcs on the overlay canvas in a dedicated `draw_topic_rings()` function called from the RAF loop's label canvas section in `graph.rs`, after the main renderer pass and before label drawing. Compute palette server-side as a `Vec<(String, [u8; 3])>` once per analysis run, persist to SurrealDB migration 8, and attach to `GraphData`.

### Canvas2D Arc Rendering Pattern

The overlay canvas uses DPR scaling. All arc coordinates must use **world space** transformed through the viewport, **not** screen coordinates. The pattern established by label drawing in `graph.rs` (lines 673+) applies:

```rust
// In the RAF loop label canvas section (graph.rs), before label drawing:
ctx.set_transform(dpr, 0.0, 0.0, dpr, 0.0, 0.0).unwrap();
// ... then topic rings use viewport.world_to_screen(node.x, node.y) for position
// and node.radius * viewport.scale for screen-space radius
```

The existing Canvas2D renderer (`canvas_renderer.rs`) draws fills and borders in **world space** (viewport transform is applied to the 2D context). The overlay canvas in `graph.rs` uses a different approach: `set_transform(dpr, 0.0, 0.0, dpr, 0.0, 0.0)` then `world_to_screen()` calls. Topic arcs on the overlay canvas must follow the **overlay pattern**, not the canvas_renderer pattern.

Key difference for the Canvas2D renderer path: `canvas_renderer.rs` draws node borders as part of the same pass as node fills. For the Canvas2D renderer, topic arcs should still be drawn on the **overlay canvas** (D-08 says "Canvas2D overlay (same canvas used for labels)"), not inline in canvas_renderer.rs. This keeps the rendering path consistent between WebGL2 and Canvas2D renderer choices.

### Arc Segment Drawing Convention

Confirmed from UI-SPEC and Claude's Discretion resolution:

```rust
const START_ANGLE: f64 = -std::f64::consts::FRAC_PI_2; // -π/2 = 12 o'clock
const TAU: f64 = std::f64::consts::TAU;
const MIN_SCREEN_RADIUS_FOR_RINGS: f64 = 6.0;

fn draw_topic_arcs(
    ctx: &web_sys::CanvasRenderingContext2d,
    sx: f64, sy: f64,      // screen-space node center
    screen_radius: f64,    // node.radius * viewport.scale
    top_keywords: &[(String, f32)],
    palette: &[PaletteEntry],  // keyword→color lookup
    dim_alpha: f64,
) {
    if screen_radius < MIN_SCREEN_RADIUS_FOR_RINGS {
        return; // plain border handled by existing code path
    }
    if top_keywords.is_empty() {
        // Dashed ring for unanalyzed nodes
        draw_dashed_ring(ctx, sx, sy, screen_radius, dim_alpha);
        return;
    }

    let score_sum: f32 = top_keywords.iter().map(|(_, s)| s).sum::<f32>().min(1.0);
    let mut current_angle = START_ANGLE;

    // Draw each keyword arc
    for (keyword, score) in top_keywords.iter().take(3) {
        let arc_angle = TAU * (*score / score_sum) as f64;
        let end_angle = current_angle + arc_angle;
        let color = palette_color_for(keyword, palette);
        ctx.save();
        ctx.set_global_alpha(dim_alpha);
        ctx.set_stroke_style_str(&color);
        ctx.set_line_width(3.0);
        ctx.begin_path();
        ctx.arc(sx, sy, screen_radius, current_angle, end_angle).unwrap();
        ctx.stroke();
        ctx.restore();
        current_angle = end_angle;
    }

    // Neutral remainder arc
    let end = START_ANGLE + TAU;
    if current_angle < end - 0.01 {
        ctx.save();
        ctx.set_global_alpha(dim_alpha);
        ctx.set_stroke_style_str("#3a424c");
        ctx.set_line_width(3.0);
        ctx.begin_path();
        ctx.arc(sx, sy, screen_radius, current_angle, end).unwrap();
        ctx.stroke();
        ctx.restore();
    }
}
```

**Critical:** `ctx.arc()` in `web_sys` takes `x, y, radius, start_angle, end_angle` and draws counterclockwise by default. To draw clockwise (the expected direction), note that `web_sys` Canvas2D `arc` has `anticlockwise: bool` defaulting to `false` (clockwise). Use the 5-argument version. The 6th argument `anticlockwise` defaults to `false` in browsers = clockwise.

Source: `web_sys::CanvasRenderingContext2d::arc` signature — 5 required args (x, y, radius, start, end), 6th is optional anticlockwise (default false = clockwise). Confidence: HIGH (web_sys docs + Canvas2D spec).

### Integration Point: is_dimmed Logic

The existing `is_dimmed` closure in `canvas_renderer.rs` (and equivalent in the WebGL2 path via `GraphState`) currently dims based on graph selection neighborhood. The keyword filter adds a second dimension to dimming.

The cleanest integration: pass `active_topic_filter: &HashSet<String>` into the `draw_topic_rings()` function and compute a composite dim alpha:

```rust
let topic_dimmed = !active_topic_filter.is_empty()
    && !node.top_keywords.iter().any(|(kw, _)| active_topic_filter.contains(kw));
let arc_alpha = if graph_dimmed || topic_dimmed { 0.3 } else { 1.0 };
```

The `active_topic_filter` signal is `RwSignal<HashSet<String>>` read via `get_untracked()` in the RAF loop (same pattern as `show_contradictions`, `label_mode`, etc.).

**Note:** The node fill dimming (existing `is_dimmed` in `canvas_renderer.rs`) should also be extended to dim when `topic_dimmed`. This requires either:
- Passing `active_topic_filter` into the renderer, OR
- Adding a `topic_dimmed: bool` flag to `NodeState`

Adding to `NodeState` is cleaner (keeps renderer stateless) — set it in the RAF loop alongside LOD/temporal visibility updates.

### Palette Computation (Server-Side)

The palette is built from `paper_analysis` table data already in SurrealDB. Algorithm:

1. Load all `PaperAnalysis` records (existing `get_all_analyses()` in `AnalysisRepository`).
2. For each unique keyword across all papers, compute inter-paper TF-IDF variance.
3. Rank by variance descending; take top 8.
4. Assign evenly-spaced OKLCH hues: `h = slot_index * (360 / 8)`, L=0.72, C=0.18.
5. Convert OKLCH→sRGB via manual computation (no external crate needed — the UI-SPEC provides pre-approximated hex values for 8 slots).

For OKLCH to sRGB conversion server-side: the UI-SPEC provides the final hex values as the target. The palette computation can use the pre-specified hex table for 8 slots and simply assign keywords to slots by variance rank, without implementing a full OKLCH conversion. This eliminates all color math complexity.

```rust
const PALETTE_COLORS: &[[u8; 3]] = &[
    [0xe8, 0xa3, 0x4b], // slot 0: amber
    [0xb8, 0xcc, 0x52], // slot 1: yellow-green
    [0x56, 0xc7, 0x6b], // slot 2: green
    [0x40, 0xc9, 0xa4], // slot 3: teal
    [0x58, 0xb4, 0xe8], // slot 4: cyan-blue
    [0x88, 0x84, 0xe0], // slot 5: blue-violet
    [0xcc, 0x6f, 0xd6], // slot 6: magenta
    [0xe0, 0x60, 0x80], // slot 7: rose
];
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Color space conversion | Custom OKLCH→sRGB math | Pre-computed hex table (8 entries) | UI-SPEC already specifies exact hex values; computation is unnecessary complexity |
| Dash reset after `set_line_dash` | Assuming it auto-resets | Always call `set_line_dash(&Array::new())` | Canvas dash state is persistent; `canvas_renderer.rs` line 167 shows the existing pattern with `ctx.restore()` doing the job |
| Per-frame palette lookup | HashMap or linear scan | Index `palette` as a `Vec`, precompute keyword→slot map at load time into a `HashMap<String, usize>` in GraphState | Per-frame linear scan over palette for every arc segment of every visible node would be O(nodes × 3 × palette_size) |
| Circular arc as polygon | Manual sin/cos polygon approximation | `ctx.arc()` directly | Canvas2D arc is anti-aliased natively; polygon approaches produce visible faceting |

## Common Pitfalls

### Pitfall 1: Canvas2D Context is Shared, State Leaks Between Draws

**What goes wrong:** After drawing dashed arcs, subsequent strokes (labels, planetary rings) inherit the dash pattern, producing dashed labels.
**Why it happens:** Canvas2D state (line dash, line width, stroke style, global alpha) persists until explicitly reset. `ctx.restore()` only restores state if `ctx.save()` was called.
**How to avoid:** Every arc drawing block must wrap in `ctx.save()` / `ctx.restore()`. Reset `set_line_dash(&Array::new())` explicitly before `restore()` or rely on `restore()` if the save predates the `set_line_dash` call. The existing ABC-bridge code in `canvas_renderer.rs` uses `ctx.save()` + `ctx.restore()` — follow this exact pattern.
**Warning signs:** Labels appear dashed; planetary ring appears dashed.

### Pitfall 2: Overlay Canvas vs Main Canvas Coordinate Systems

**What goes wrong:** Topic arcs drawn in world coordinates appear offset from node centers.
**Why it happens:** The overlay canvas does NOT have `viewport.apply()` called on it (unlike `canvas_renderer.rs`). It uses `set_transform(dpr, 0.0, 0.0, dpr, 0.0, 0.0)` then explicit `world_to_screen()` calls.
**How to avoid:** Use `viewport.world_to_screen(node.x, node.y)` to get screen coords, then `node.radius * viewport.scale` for screen-space radius. Do NOT call `viewport.apply()` on the overlay canvas context.
**Warning signs:** Arcs appear in wrong position relative to node fills, especially at non-default zoom.

### Pitfall 3: Arc Line Width is Screen-Space, Not World-Space

**What goes wrong:** On the main canvas renderer, `canvas_renderer.rs` uses `self.ctx.set_line_width(1.0 / viewport.scale)` to keep borders 1px regardless of zoom. On the overlay canvas, the transform is DPR-only (no viewport scale), so `set_line_width(3.0)` means 3 CSS pixels.
**Why it happens:** Different coordinate system conventions between the two canvas elements.
**How to avoid:** On the overlay canvas, `set_line_width(3.0)` is correct for 3px visual width (DPR is handled by the transform). Do NOT divide by `viewport.scale` for overlay canvas arcs.
**Warning signs:** Ring width changes with zoom level.

### Pitfall 4: Palette Not Available Until Analysis Has Run

**What goes wrong:** On a fresh install with only crawl data (no analysis), `GraphData.palette` is empty. Code that indexes into palette unconditionally panics or renders garbage.
**Why it happens:** Palette computation depends on `paper_analysis` table data which only exists after TF-IDF analysis runs.
**How to avoid:** Guard all palette lookups: `if palette.is_empty() { return; }` before the toggle renders. The UI-SPEC specifies the legend section is hidden when palette is empty — implement this as a conditional in the Leptos view using `move || !palette.get().is_empty()`.
**Warning signs:** Server function returns error on fresh database; legend section visible with no entries.

### Pitfall 5: `get_graph_data()` Server Function: Palette Load Failure Must Not Block Graph

**What goes wrong:** Palette SurrealDB query fails (e.g., migration not applied, empty table), causing the entire graph load to fail.
**Why it happens:** Overly strict error propagation.
**How to avoid:** Palette load in `get_graph_data()` should fall back gracefully: `get_palette().await.unwrap_or_default()`. An empty palette means topic rings are silently disabled.

### Pitfall 6: `top_keywords` Has 5 Entries; Arcs Only Use Top 3

**What goes wrong:** All 5 keywords get arcs, covering more than the intended ring space.
**Why it happens:** `get_graph_data()` already takes top 5 from analysis (`take(5)` at line 71). Arc rendering must further limit to top 3.
**How to avoid:** `top_keywords.iter().take(3)` in arc drawing. The UI-SPEC explicitly specifies 3 arcs. This is different from the label rendering which uses top 2 pills.

### Pitfall 7: Leptos `HashSet<String>` Signal Requires Explicit Clone

**What goes wrong:** `active_topic_filter.update(|set| set.insert(kw))` where `kw` is a `String` moved into a closure that also captures `palette` — borrow checker error.
**Why it happens:** Leptos signal closures in `view!` macros require `move` captures; `String` values referenced in multiple closures must be cloned before capture.
**How to avoid:** Clone the keyword string before capture: `let kw = entry.keyword.clone(); move |_| { active_topic_filter.update(|s| { s.insert(kw.clone()); }); }`.

## Open Questions

1. **Which renderer path draws the plain 1px border for below-threshold nodes?**
   - What we know: Currently `canvas_renderer.rs` draws borders; `webgl_renderer.rs` has a built-in border effect via shader (lines 56-59, a brightness boost at node edge).
   - What's unclear: For below-threshold nodes in WebGL2 mode, the shader border is always visible. Is the 1px plain border (D-13) only relevant for Canvas2D renderer path?
   - Recommendation: Below-threshold plain border handling goes on the overlay canvas for consistency between renderers. Skip the below-threshold branch in overlay if the WebGL2 shader already provides the border effect; only apply it in Canvas2D renderer mode, or simply let the WebGL2 shader border remain as-is (it's subtle, not the `#7cb8ff` color).

2. **Palette population timing: compute in `get_graph_data()` or separate server function?**
   - What we know: `get_graph_data()` already loads all analyses and builds analysis_map (lines 61-77). Palette computation can be added inline.
   - What's unclear: Adding palette computation to `get_graph_data()` increases graph load time; a separate endpoint would allow lazy loading.
   - Recommendation: Inline in `get_graph_data()` with a graceful fallback. The palette computation is cheap (sort 8 variance values from existing analysis data). The analysis data is already loaded for `top_keywords`, so no extra DB round trip is needed.

3. **`NodeState.topic_dimmed` vs passing filter into renderer?**
   - What we know: `GraphState` is passed to `renderer.draw()` as `&GraphState`. Adding `topic_dimmed: bool` to `NodeState` requires setting it each frame in the RAF loop.
   - Recommendation: Add `topic_dimmed: bool` to `NodeState` (default false). Set in the RAF loop alongside LOD/temporal updates. This is the cleanest pattern — renderers remain stateless.
