**Wire start_crawl to CrawlQueue (resyn-app/src/server_fns/papers.rs):**

Replace the start_crawl TODO placeholder with real crawl dispatch. Per user decision "Server function triggers crawl logic on backend":

Inside the `#[cfg(feature = "ssr")]` block of `start_crawl`:
1. Retrieve `Arc<Db>` from context (same pattern as other server fns)
2. Validate paper_id format using `resyn_core::validation::validate_arxiv_id(&paper_id)`
3. Create `CrawlQueueRepository::new(&db)` and call `enqueue_if_absent(&paper_id, &paper_id, 0, max_depth)` to seed the crawl queue with the initial paper
4. Spawn a background tokio task that runs the crawl loop. The crawl loop logic should mirror what `resyn-server/src/commands/crawl.rs::run()` does after queue seeding:
   - Create the appropriate PaperSource based on the `source` string ("arxiv" or "inspirehep")
   - Run the queue-driven crawl loop (claim_next_pending, fetch, enqueue references, mark_done/mark_failed)
   - The spawned task runs independently — the server function returns immediately with "Crawl started"
5. Return `Ok(format!("Crawl started for paper {} (depth {})", paper_id, max_depth))`

**Important:** The crawl logic execution happens in a spawned tokio task so the server function responds immediately. The SSE /progress endpoint (from Phase 7) will report progress as the background crawl runs. If the crawl loop logic is complex to extract from crawl.rs, a simpler approach is acceptable: just enqueue the seed paper and return — if the serve command also runs a background queue consumer loop, the crawl will proceed. Check how the existing crawl infrastructure works and choose the approach that integrates cleanly.
