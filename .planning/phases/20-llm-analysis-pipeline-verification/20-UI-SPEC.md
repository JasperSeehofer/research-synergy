## Component Inventory

New components introduced in this phase. All must follow existing class naming and CSS variable conventions.

### 1. LLM Warning Banner (`analysis_controls.rs`)

**When shown:** Server-side: when `RESYN_LLM_PROVIDER` is unset at request time.
**Layout:** Full-width banner below `.page-title`, above dashboard cards. Not a modal. Not blocking.
**Structure:**
```
[warning icon] "LLM provider not configured — showing NLP-only results.
               Set RESYN_LLM_PROVIDER for full analysis."
```
**CSS class:** `warning-banner` (new). Left border `4px solid var(--color-bridge)`, background `var(--color-surface)`, padding `var(--space-md)`, radius `var(--radius-md)`, text `var(--font-size-body)`.

### 2. Run Analysis Button (inline on dashboard)

**Label:** "Run Analysis" (idle) / "Analysis running…" (pending, disabled).
**CSS class:** `btn-primary` — reuse existing class, no new CSS needed.
**Placement:** Below dashboard summary cards. Full width of the dashboard content column is wrong — use natural button width, left-aligned, `margin-top: var(--space-lg)`.
**State transitions:**
- Idle: "Run Analysis" — enabled.
- Pending (SSE `analysis_*` events active): "Analysis running…" — `disabled` attribute.
- Complete (SSE `analysis_complete`): button re-enables; panels auto-refetch.

### 3. Post-Crawl Inline Prompt

**When shown:** After SSE emits `event_type == "complete"` for a crawl, and before analysis has been triggered.
**Layout:** Inline below the crawl progress summary in the sidebar footer. Single line.
**Structure:**
```
"Analysis available —" [Run now] button
```
**CSS:** `analysis-prompt` (new). `font-size: var(--font-size-label)`, `color: var(--color-text-muted)`. The inline "Run now" button uses `color: var(--color-accent)`, `background: none`, `border: none`, `font-size: var(--font-size-label)`, `cursor: pointer`.
**Behavior:** Clicking "Run now" dispatches `start_analysis` server action. Prompt hides once analysis SSE events begin.

### 4. Analysis Progress Display (sidebar footer)

**When shown:** While SSE emits `event_type` values of `analysis_extracting`, `analysis_nlp`, `analysis_llm`, `analysis_gaps`.
**Layout:** Replaces the crawl progress display in the sidebar footer while analysis is running. Same region, same visual structure.
**Structure:**
```
[progress bar — accent fill]
"Analysis: [stage label]"   [papers_found] / [papers_pending]
```
**Stage labels** (map from `event_type`):
- `analysis_extracting` → "Extracting text"
- `analysis_nlp` → "Running TF-IDF"
- `analysis_llm` → "Annotating papers"
- `analysis_gaps` → "Finding gaps"
**CSS:** Reuse `.crawl-progress-bar`, `.crawl-progress-fill`, `.crawl-progress-stats`. New class `analysis-stage-label` mirrors `.crawl-progress-running` — `font-size: var(--font-size-label)`, `color: var(--color-text-muted)`, `margin-bottom: var(--space-sm)`.

### 5. Empty State with CTA (result panels — D-04)

**When shown:** All three result panels (gaps, open problems, methods) when data is empty because analysis has not run.
**Layout:** Centered within the panel content area. Heading + body + button stacked vertically with `gap: var(--space-md)`.
**Structure:**
```
"No analysis results yet"          [heading — .empty-state-heading]
"Run analysis to see [panel name] here."  [body — .empty-state-body]
[Run Analysis]                     [button — .btn-primary]
```
**CSS:** Reuse existing `.empty-state`, `.empty-state-heading`, `.empty-state-body` classes. The CTA button dispatches `start_analysis` via the same action used on the dashboard.
**Distinction from filter-empty state:** Filter-empty state (data exists but filters exclude all) uses `.empty-state-body` without a CTA button — existing pattern in `gaps.rs` covers this.

Source: CONTEXT.md D-04; existing `gaps.rs`, `open_problems.rs`, `dashboard.rs` component patterns.

## Interaction States

Every interactive element introduced in this phase must implement all four states:

| Element                    | Default                      | Hover                              | Active/Pressed           | Disabled                            |
|----------------------------|------------------------------|------------------------------------|--------------------------|-------------------------------------|
| "Run Analysis" btn         | `btn-primary` filled         | inherited hover from `.btn-primary`| opacity 0.9              | `opacity: 0.5; cursor: not-allowed` |
| "Run now" inline link-btn  | `--color-accent` text, no bg | underline                          | opacity 0.8              | hidden (not shown when disabled)    |
| Analysis progress bar      | fills left-to-right (accent) | no hover interaction               | n/a                      | n/a                                 |

**Focus:** All buttons must show `outline: 2px solid var(--color-accent); outline-offset: 2px` on `:focus-visible`. This is already the global pattern for inputs — apply the same rule to buttons that receive keyboard focus.

## SSE Event → UI State Mapping

This phase introduces new `event_type` values on the existing `/progress` SSE stream. The UI must interpret them as follows:

| SSE `event_type`         | Dashboard sidebar state    | Result panel state                                    |
|--------------------------|----------------------------|-------------------------------------------------------|
| `analysis_extracting`    | Show analysis progress bar | No change — panels show previous data or empty state  |
| `analysis_nlp`           | Show analysis progress bar | No change                                             |
| `analysis_llm`           | Show analysis progress bar | No change                                             |
| `analysis_gaps`          | Show analysis progress bar | No change                                             |
| `analysis_complete`      | Hide progress, show idle   | Trigger `Resource::refetch()` on all three panels     |
| `analysis_error`         | Hide progress, show error  | No change — panels remain in previous state           |

**"is_running" logic extension:** The existing `is_running` check (`event_type != "complete" && event_type != "idle"`) will naturally include analysis stage events — no additional logic needed beyond adding the `analysis_*` string variants.

Source: CONTEXT.md D-03, D-05; RESEARCH.md Pattern 2, Pattern 3; `crawl_progress.rs` lines 38–44.
