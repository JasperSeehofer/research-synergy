<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI flag parsing | User-supplied CLI args reach internal config; clap validates types but values are otherwise untrusted |
| Worker-loop control flow | New conditional block must not break existing reference-fetching path |
| Enqueue path | Newly discovered arXiv IDs (from S2 forward citations) become BFS queue entries — could amplify crawl scope |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-16 | D (DoS) — `--max-forward-citations 0` cap-bypass | clap parser + `with_max_forward_citations(0)` | accept | Cap=0 means "fetch nothing meaningful". `fetch_citing_papers_inner` (plan 01) checks `all_citations.len() >= cap` after each page; with cap=0 the check fires after the first page (which may have up to 500 items, then truncate to 0). Worst case: one HTTP call + no edges. No DoS amplification |
| T-28-17 | D (DoS) — `--max-forward-citations` very large value (e.g. usize::MAX) | clap default+user override | mitigate | Pagination is still bounded by S2's actual citation count for the paper (real ceiling ~10^4 for highly-cited seeds). Combined with the BFS `--max-depth` and the queue-deduplication via `enqueue_if_absent`, the total enqueue count is bounded. No mitigation beyond documentation |
| T-28-18 | T (Tampering) — edge-injection via crafted S2 response with malicious arxiv_id | upstream `convert_s2_paper_to_ref` -> `Reference.arxiv_eprint` -> `paper_record_id` | mitigate | The arxiv_id ultimately reaches `paper_record_id(&id)` which builds a typed `RecordId` (not a string-interpolated query). SurrealDB record IDs are validated by the driver. Even if a malicious value passes, it can only create dangling edges in the `cites` table — no privilege escalation, no data exfiltration |
| T-28-19 | I (Information Disclosure) — log injection via paper.id, source name | `tracing::warn!(paper_id = ..., source = ...)` | mitigate | All `tracing` fields use structured logging (key-value), not format-string interpolation. Subscriber escapes special characters. `paper.id` is pre-validated by `validate_arxiv_id` before reaching the worker |
| T-28-20 | E (Elevation of Privilege) — `--bidirectional` enabling unintended write paths | New worker-loop block + `upsert_inverse_citations_batch` | accept | The new method writes only to the `cites` table (same as existing `upsert_citations`). No new authorization surface. The flag is opt-in |
| T-28-21 | T (Tampering) — `paper.citing_papers` populated from a malicious upstream JSON before the worker reaches `fetch_citing_papers` | Paper deserialization in any path that loads JSON-imported papers | accept | The worker calls `source.fetch_paper(&entry.paper_id)` which constructs a fresh `Paper` from the S2 paper endpoint — `citing_papers` is empty by default. Then `fetch_citing_papers` overwrites the field with the new fetch result. Pre-population from a malicious deserialization path cannot survive the overwrite |
| T-28-22 | D (DoS) — runaway enqueue from `enqueue_if_absent` loop | New enqueue block in worker | mitigate | `enqueue_if_absent` already deduplicates by arxiv_id (existing behaviour). The depth-cap (`--max-depth`) prevents enqueue at `depth+1 > max_depth` from being claimed. Combined with `--max-forward-citations`, the per-paper enqueue ceiling is bounded |
</threat_model>
