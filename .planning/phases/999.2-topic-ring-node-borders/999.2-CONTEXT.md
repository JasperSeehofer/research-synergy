# Phase 999.2: Topic Ring Node Borders - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Encode each node's top-3 TF-IDF keywords as colored arc segments on its border ring, creating a visual topic fingerprint. Global corpus keywords each get a fixed color (variance-ranked). Arc length proportional to normalized TF-IDF score. Includes a keyword→color legend panel in the graph controls sidebar with click-to-filter interaction, and an independent "Topic rings" toggle.

</domain>

<decisions>
## Implementation Decisions

### Color Palette Strategy
- **D-01:** Global palette built from top-N discriminative keywords ranked by inter-paper TF-IDF variance. High variance = discriminative (not ubiquitous). Keywords beyond the palette use neutral gray.
- **D-02:** Palette size: 8-10 distinct perceptually distinguishable hues.
- **D-03:** Palette cached in SurrealDB for consistency across sessions. Keyword→color mapping persisted.
- **D-04:** Palette invalidated and recomputed only on new crawl (not on re-analysis). Stable within a session.

### Ring Rendering Approach
- **D-05:** Topic arcs replace the existing node border stroke. The border IS the topic ring. No extra layer — 3px colored arcs instead of 1px neutral border.
- **D-06:** Seed node keeps its planetary ring on top of topic arcs: [node fill] → [3px topic arcs] → [gap] → [planetary ring].
- **D-07:** Remaining circumference (where TF-IDF scores sum < 1.0) rendered in neutral/default border color.
- **D-08:** WebGL2 renderer: topic arcs drawn on the Canvas2D overlay (same canvas used for labels), not via shader changes. WebGL2 draws filled circles only; Canvas2D draws arc borders on top.
- **D-09:** Topic rings controlled by an independent toggle ("Topic Rings" checkbox) in graph controls — visible in any label mode, not tied to Keywords label mode.

### Legend Panel & Interaction
- **D-10:** Legend placed inside the existing graph controls sidebar as a "Topic Colors" section, below the label mode dropdown and topic rings toggle.
- **D-11:** Each legend entry shows a color swatch and keyword name. Scrollable if palette exceeds visible space.
- **D-12:** Click-to-filter: clicking a keyword in the legend dims all nodes that DON'T have that keyword in their top-3. Click again to clear. Multiple keywords can be active (union — any matching keyword keeps node bright).

### Small Node Threshold
- **D-13:** Below a minimum screen-space radius (e.g. 6px), hide topic arcs and show plain 1px border instead. Screen-space = world radius × viewport scale — adapts to zoom level automatically.
- **D-14:** Ring width fixed at 3px regardless of node size (above the threshold).
- **D-15:** Unanalyzed papers (no TF-IDF data) show a dashed neutral ring to indicate "topic data missing." Visually distinct from analyzed nodes with topic arcs and from the plain border of below-threshold nodes.

### Claude's Discretion
- Exact minimum screen-space radius threshold for topic ring visibility (suggested 6px, can tune)
- Perceptual hue selection algorithm (e.g. evenly spaced in OKLCH, or curated palette)
- Dash pattern for unanalyzed node rings
- Canvas2D arc drawing order and anti-aliasing approach
- DB schema design for palette storage (new table vs field on existing table)
- Arc start angle convention (top of circle = 0, clockwise)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `GraphNode.top_keywords: Vec<(String, f32)>` — keyword data already flows from server to WASM, no new server function needed for per-node data
- `get_graph_data()` already joins paper data with analysis data — palette computation can be added here or as a separate server function
- Canvas2D overlay canvas — already used for label pills and text, can draw arc borders here for both renderers
- `ctx.arc(x, y, radius, start_angle, end_angle)` — Canvas2D API directly supports arc segments
- Graph controls sidebar — existing Leptos component with dropdowns and toggles

### Established Patterns
- `NodeState` holds per-node rendering data, constructed from `GraphNode` in `layout_state.rs`
- `GraphNode` serialized from server function, deserialized in WASM — new fields (palette data) cross the SSR/WASM boundary
- Node rendering: Canvas2D draws filled circle + border stroke; WebGL2 draws filled circle via instanced quads
- Seed node distinction: amber fill + planetary ring — topic arcs must coexist without conflicting

### Integration Points
- `GraphData` needs corpus-level palette data (keyword→color mapping) alongside per-node keywords
- `NodeState` may need precomputed arc segment data (angles, colors) for per-frame rendering
- Canvas2D overlay `draw_nodes` or new `draw_topic_rings` function needed
- Graph controls sidebar needs "Topic Rings" toggle + "Topic Colors" legend section
- SurrealDB schema needs palette storage table/field
- DB migration for palette persistence

</code_context>

<specifics>
## Specific Ideas

- User wants topic rings as an independent toggle, not coupled to label mode — rings show "what" visually while labels provide textual detail
- Dashed ring for unanalyzed papers — explicit visual cue that topic data is missing, not just absence of color
- Dim-based filtering via legend clicks — matching nodes stay bright, non-matching dim, multiple keywords union
- Palette cached in DB for cross-session consistency — not recomputed every page load

</specifics>
